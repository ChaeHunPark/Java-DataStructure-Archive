# 📅 WEEK 4: 정렬 알고리즘 & 문자열 탐색 (Sorting & Trie)

4주차에는 데이터 정렬의 효율성을 결정짓는 **시간 복잡도($O$)**의 실질적 차이를 벤치마킹하고, 문자열 탐색에 최적화된 **트라이(Trie)** 자료구조를 학습했습니다. 특히 퀵 정렬의 파티션 로직을 직접 시뮬레이션하며 알고리즘의 내부 동작 원리를 깊이 있게 분석했습니다.

## 📌 학습 로드맵
* **$O(n^2)$ 정렬**: 선택(Selection), 삽입(Insertion), 버블(Bubble) 정렬의 메커니즘 이해 및 구현
* **$O(n \log n)$ 정렬**: 퀵(Quick) 정렬의 분할 정복(Divide & Conquer)과 병합(Merge) 정렬의 안정적 성능 분석
* **성능 벤치마킹**: 람다(Lambda)를 활용한 소요 시간 측정 유틸리티 설계 및 알고리즘별 성능 격차 실증
* **트라이(Trie)**: 문자열의 각 문자를 노드로 구성하여 검색 효율을 $O(L)$(단어 길이)로 극대화하는 구조 학습

## 📊 알고리즘별 성능 벤치마크 (n=5,000)
| 분류 | 알고리즘 | 평균 소요 시간 | 특징 |
| :--- | :--- | :--- | :--- |
| **기초** | 버블 (Bubble) | **약 45.1 ms** | 인접 요소 교환, 가장 낮은 효율 ($O(n^2)$) |
| **기초** | 선택 (Selection) | 약 28.5 ms | 최솟값 탐색 후 교환, 교환 횟수 최적화 |
| **기초** | 삽입 (Insertion) | 약 18.2 ms | 정렬된 구역에 '비집고 들어가기', 최선의 경우 $O(n)$ |
| **심화** | **퀵 (Quick)** | **약 3.1 ms** | **평균 15배 이상 빠름.** 분할 정복의 압도적 효율 |
| **심화** | 병합 (Merge) | 약 4.5 ms | 추가 메모리를 사용하나 어떤 상황에서도 안정적 성능 |

## 💡 주요 기술적 의사결정 (Key Decisions)

### 1. 퀵 정렬의 '편 가르기' 로직 최적화 (sorting/)
피벗(Pivot)을 기준으로 데이터를 나누는 `partition` 메서드에서 경계선 포인터(`i`)와 탐사대(`j`)의 역할을 명확히 분리했습니다. `i`를 통해 '작은 값들의 영토'를 확장해 나가는 로직을 직접 시뮬레이션하며, 불필요한 연산을 줄이고 제자리 정렬(In-place Sort)의 메모리 효율성을 확보했습니다.

### 2. 병합 정렬의 분할 후 조립 전략
퀵 정렬과 달리 **"선 분할 후 정렬"** 방식을 취하는 병합 정렬을 구현했습니다. 원소가 1개가 될 때까지 재귀적으로 쪼갠 뒤, `merge` 과정에서 두 배열을 포인터로 비교하며 합치는 방식을 통해 데이터 상태에 휘둘리지 않는 $O(n \log n)$의 안정성을 확보했습니다.

### 3. 유연한 성능 측정을 위한 함수형 인터페이스 활용
각 정렬 메서드의 성능을 객관적으로 비교하기 위해 `measureTime(String label, Runnable task)` 메서드를 설계했습니다. 자바의 **람다(Lambda)**를 인자로 전달받아 측정 로직과 알고리즘 로직을 분리함으로써, 테스트 코드의 가독성을 높이고 반복적인 측정을 용이하게 만들었습니다.

### 4. 문자열 검색의 혁신: Trie (trie/)
수만 개의 단어 속에서 특정 키워드를 찾을 때, `contains()`와 같은 선형 탐색($O(n \times L)$) 대신 **트라이(Trie)** 구조를 도입했습니다. 메모리를 더 사용하는 대신 검색 시간을 단어의 길이에만 의존하게 만들어 대규모 텍스트 처리의 병목을 해결했습니다.
