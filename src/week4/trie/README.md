## 📅 Week 4: Trie (Prefix Tree) - 문자열 탐색의 최적화와 접두사 사전

### 1. 핵심 개념 리마인드
* **트라이(Trie)**: 문자열의 각 문자를 노드로 저장하여, 루트에서 바닥까지의 경로가 하나의 단어가 되는 **문자열 특화 트리** 구조.
* **접두사 공유(Prefix Sharing)**: 'apple'과 'apply'처럼 앞부분이 같은 단어들은 'appl'까지의 노드를 공유하여 메모리 효율과 탐색 속도를 동시에 잡음.
* **시간 복잡도 혁신**: 단어의 개수가 수억 개라도, 탐색 속도는 전체 개수($N$)가 아닌 **찾고자 하는 단어의 길이($L$)**에만 비례함 ($O(L)$).

### 3. 주요 로직 구현 포인트
* **노드 설계 (TrieNode)**: `Map<Character, TrieNode> children`을 통해 자식 노드들을 관리하며, 단어의 끝을 알리는 `isEndOfWord` (깃발) 필드가 핵심.
* **삽입(Insert)**: 루트부터 내려가며 문자에 해당하는 노드가 없으면 새로 생성하고, 단어가 끝나면 반드시 깃발을 꽂아줌.
* **삭제(Remove)의 역주행 (Backtracking)**:
    - 단순히 깃발만 내리는 것이 아니라, 재귀를 통해 단어 끝부터 거꾸로 올라오며 **"나를 필요로 하는 다른 단어가 없는가?"**를 체크.
    - 자식이 없고 깃발도 없는 '막다른 길' 노드는 `null` 처리를 통해 메모리에서 물리적으로 제거함.

### 4. 시간 복잡도 분석 ($O$)
| 연산 | 복잡도 | 이유 |
| :--- | :--- | :--- |
| **Insert** | $O(L)$ | 단어의 각 문자($L$)를 하나씩 노드로 생성하며 내려감 |
| **Search** | $O(L)$ | 단어의 길이만큼 경로를 따라가며 마지막 노드의 깃발 확인 |
| **Delete** | $O(L)$ | 단어 끝까지 탐색 후, 다시 루트 방향으로 올라오며 노드 정리 |

> **주의**: 알파벳 26개를 배열(`new TrieNode[26]`)로 선언할 경우 탐색은 빠르나 메모리 낭비가 심함. 실무나 복잡한 문제에서는 `HashMap`을 사용하여 필요한 문자 노드만 생성하는 방식을 선호함.

### 💡 Insight
"이진 탐색 트리(BST)가 데이터의 크기 비교를 통해 범용적인 탐색을 수행한다면, **트라이(Trie)는 '문자열'이라는 데이터 특성을 극한으로 활용한 구조다.** 특히 삭제 로직에서 재귀의 리턴값을 활용해 하위 노드부터 차례대로 철거 여부를 결정하는 과정은, '필요 없는 경로는 과감히 정리한다'는 자원 관리의 정수를 보여준다. 백준 5052번과 같은 접두어 문제를 풀 때 `contains`의 $O(N^2)$ 늪에서 벗어나 $O(L)$의 세계로 진입하게 해주는, 문자열 처리의 가장 강력한 무기 중 하나다."
