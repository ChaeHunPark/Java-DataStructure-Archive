# 📅 Week 4: Sorting (정렬) - 알고리즘 효율의 분기점 $O(n^2)$ vs $O(n \log n)$

### 1. 핵심 개념 리마인드
* **$O(n^2)$ 계열 (기초)**:
    - **선택(Selection)**: 전체에서 가장 작은 수를 찾아 정렬 안 된 부분의 맨 앞과 교체.
    - **삽입(Insertion)**: 두 번째 요소부터 시작하여 왼쪽의 정렬된 구역 내 적절한 위치를 찾아 삽입.
    - **버블(Bubble)**: 인접한 두 수를 비교하여 큰 수를 뒤로 밀어내는 방식.
* **$O(n \log n)$ 계열 (심화)**:
    - **퀵 정렬(Quick)**: **'편 가르기(Partition)'** 전략. 피벗(Pivot)을 기준으로 작은 값과 큰 값을 나누며 재귀적으로 분할 정복.
    - **병합 정렬(Merge)**: **'일단 쪼개기'** 전략. 원소가 1개가 될 때까지 분할한 뒤, 다시 합치는 과정에서 정렬 수행.

### 2. 주요 로직 구현 포인트
* **퀵 정렬: 작은 수의 구역 경계선(`i`)**
    - 탐사대(`j`)가 피벗보다 작은 값을 발견하면 경계선 `i`를 확장하고 그 자리에 값을 소환(`swap`).
    - 루프 종료 후 `i + 1` 자리에 피벗을 안착시켜 피벗의 최종 위치를 확정하는 것이 핵심.
* **병합 정렬: 포인터 관리(`i, j, k`)**
    - 왼쪽(`leftArr`)과 오른쪽(`rightArr`) 임시 배열의 포인터를 비교하여 원본 배열(`k`)에 순차적으로 삽입.
    - 한쪽 배열이 먼저 소모되면 남은 요소를 그대로 복사하는 예외 처리(`while`문)가 필수적.
* **성능 측정 유틸리티**:
    - `Runnable` 인터페이스와 **람다식**을 활용하여 측정 로직과 정렬 로직을 완전히 분리, 코드 재사용성 및 가독성 확보.

### 3. 성능 벤치마크 분석
*랜덤 정수 5,000개 데이터 환경 테스트 결과*

| 알고리즘 | 특징 | 결과 (Rank) |
| :--- | :--- | :---: |
| **퀵 정렬** | 평균적으로 가장 빠름 (In-place) | **1위 (압도적)** |
| **병합 정렬** | 어떤 상황에서도 $O(n \log n)$ 보장 | 2위 |
| **삽입 정렬** | 정렬이 어느 정도 된 데이터에서 유리 | 3위 |
| **선택 정렬** | 데이터 교환 횟수가 적음 | 4위 |
| **버블 정렬** | 구현이 직관적이나 가장 느림 | 5위 |

### 💡 Insight
"정렬 알고리즘을 직접 구현하며 느낀 가장 큰 수확은 **'데이터를 대하는 관점의 변화'**다. 단순히 값을 순서대로 나열하는 것을 넘어, 퀵 정렬의 피벗처럼 기준을 어떻게 잡느냐에 따라 연산 횟수가 기하급수적으로 차이 난다는 점을 체감했다.

특히 성능 테스트를 통해 $O(n^2)$ 정렬들과 퀵 정렬 사이의 **약 15배에 달하는 시간 격차**를 목격하며, 대규모 데이터를 다루는 백엔드 환경에서 왜 효율적인 알고리즘을 선택해야 하는지 그 당위성을 찾을 수 있었다. 또한, 병합 정렬처럼 메모리를 추가로 사용하여 시간을 단축하는 **Trade-off** 개념을 코드로 직접 증명해 본 유익한 과정이었다."

## 🔥 핵심 인사이트
> "퀵 정렬의 `i`와 `j`가 교차하며 구역을 나누는 과정은 복잡한 문제를 단순화하는 **분할 정복**의 정수를 보여준다. 15배의 성능 차이는 단순한 수치가 아니라, 효율적인 로직 하나가 시스템 전체의 퍼포먼스를 결정짓는다는 것을 의미한다. 결국 알고리즘의 우아함만큼이나 중요한 것은 데이터를 담는 그릇의 물리적 성질과 시간의 관계임을 이해하게 되었다."
