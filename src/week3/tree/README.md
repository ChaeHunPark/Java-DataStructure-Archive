## 📅 Week 3: Binary Search Tree (이진 탐색 트리) - 효율적 탐색과 동적 관리

### 1. 핵심 개념 리마인드
* **이진 탐색 트리(BST)**: 모든 노드가 "왼쪽 자식 < 부모 < 오른쪽 자식"의 규칙을 엄격하게 따르는 트리 구조.
* **엄격한 정렬(Strict Ordering)**: 힙(Heap)이 부모-자식 간의 느슨한 서열만 유지했다면, BST는 좌우 관계까지 명확히 정의하여 $O(\log n)$의 탐색 속도를 보장함.
* **동적 데이터 관리**: 배열과 달리 삽입과 삭제 시 데이터를 밀고 당길 필요 없이, 참조(Reference) 연결만 바꿔주는 것만으로 구조 재편이 가능함.



### 2. 주요 로직 구현 포인트
* **재귀적 구조 (Recursive Design)**: `add`, `remove` 등 모든 연산을 재귀 함수로 구현. 부모 노드가 자식의 변경된 상태(주소값)를 리턴 받아 다시 연결하는 방식 사용.
* **삭제(Remove)의 3가지 케이스**:
    - **자식 0~1개**: 한쪽 팔이 비어있다면 반대쪽 자식(혹은 null)을 부모에게 즉시 반환하여 입양시킴.
    - **자식 2개 (Successor)**: 오른쪽 서브트리의 최솟값(`findSmallestValue`)을 찾아 데이터를 덮어쓴 후, 아래에 남은 중복 노드를 제거함.
* **후계자 탐색**: `current.right`에서 시작해 `left`가 `null`일 때까지 파고들어 "나보다 크면서 가장 가까운 값"을 찾아냄.



### 3. 시간 복잡도 분석 ($O$)
| 연산 | 복잡도 | 이유 |
| :--- | :--- | :--- |
| **Search** | $O(\log n)$ | 매 단계마다 탐색 범위가 절반으로 줄어듦 (업다운 게임 원리) |
| **Add** | $O(\log n)$ | 적절한 위치를 찾기 위해 트리의 높이($h$)만큼 내려감 |
| **Remove** | $O(\log n)$ | 타겟 탐색 후 후계자를 찾는 과정이 트리 높이에 비례함 |

> **주의**: 트리가 한쪽으로 치우칠 경우(Skewed Tree) 최악의 경우 $O(n)$까지 성능이 저하될 수 있음. 이를 방지하기 위해 실무에서는 AVL 트리나 Red-Black 트리를 활용함.

### 💡 Insight
"힙(Heap)이 **최솟값 추출**이라는 특정 목적에 특화된 '효율적 타협'의 미학이라면, BST는 **데이터의 탐색과 정렬 상태 유지**라는 범용적인 효율성을 극대화한 구조다. 특히 삭제 과정에서 발생하는 '데이터 복사 후 원본 제거' 전략은 복잡한 트리 구조를 파괴하지 않고 유지하는 재귀의 정수를 보여준다. `current.right = removeRecursive(...)` 한 줄에 담긴 '수사 범위 제한'의 논리는 재귀적 사고가 도달할 수 있는 가장 우아한 해결책 중 하나다."
